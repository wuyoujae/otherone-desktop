Introduction

Otherone is a desktop-grade autonomous workspace. It gives agents access to project context, tools, review checkpoints, and persistent memory while keeping important actions visible.

The workspace is designed around a simple loop: describe a goal, let the agent plan, review the plan, approve actions, and keep the resulting context available for future work.

Designed for real work

Otherone is not just a chat window. It is a structured environment for planning, execution, tool orchestration, and human review.

Installation

Download the desktop app for your operating system, then sign in or create a local workspace. Otherone can run with local project access and optional cloud sync.

Download Otherone

Install the macOS, Windows, or Linux build from the download menu on the website.

Create a workspace

Start with a personal workspace or connect a team workspace if your organization already uses Otherone.

Attach a project

Select a local folder so the agent can inspect relevant files and generate context-aware plans.

terminal
# Optional CLI install
npm install -g otherone

# Authenticate the local workspace
otherone login

# Attach the current project
otherone workspace attach .

Run your first task

A task is a goal-driven agent run. Every task starts with a plain-language instruction and becomes a plan with visible steps.

example prompt
Review the authentication module.
Find security risks, propose fixes, and wait for approval before changing files.

The agent will scan context, produce an execution plan, identify files or tools it needs, and ask for confirmation before applying sensitive changes.

Workspace structure

A workspace contains projects, tasks, memory, tools, plugins, and review history. It is the boundary where agents understand what they can inspect and what they can modify.

Object Purpose Typical use
Project Local or remote files available to the agent. Code review, refactor, research, documentation.
Task A goal-driven run with plan, actions, and results. Fix a bug, summarize a repo, create a workflow.
Memory Persistent context from decisions and prior work. Keep long-running projects coherent.
Tool An approved capability the agent can request. Terminal, file edits, browser, plugins.

Agents

Agents are specialized execution units. Each agent can plan, inspect context, request tools, and produce structured outputs.

Common agent types

  • Architecture Agent: inspects system design, dependencies, and project structure.
  • Code Agent: proposes patches, refactors modules, and validates implementation details.
  • Research Agent: gathers and organizes information into structured summaries.
  • Workflow Agent: coordinates repeatable multi-step processes.

Memory

Memory helps Otherone preserve decisions, unresolved tasks, assumptions, and project-specific knowledge across sessions.

Memory is scoped

Workspace memory is scoped to the current workspace. Project-specific details should not leak into unrelated workspaces.

memory.json
{
  "workspace": "otherone-web",
  "decisions": [
    "Use lightweight SVG for brand assets",
    "Keep pricing cards minimal and conversion-focused"
  ],
  "open_tasks": [
    "Finalize docs page",
    "Connect changelog entries to release data"
  ]
}

Tools

Tools are capabilities agents can request. Otherone keeps tool calls visible so you can inspect what the agent wants to do before it acts.

Tool permissions

Every tool can be configured as read-only, approval required, or auto-run. Sensitive operations should remain approval-based.

Tool Permission Description
File system Approval required Read, create, or modify files inside approved project folders.
Terminal Approval required Run commands and inspect output with review history.
Browser Read-only Inspect pages or UI state during research and testing.
Plugins Configurable Connect external services and custom workflows.

Review model

Otherone uses a review-first model. The agent can propose plans and request actions, but important changes remain visible and reversible.

  1. The agent creates a plan.
  2. The workspace shows requested tools and affected files.
  3. You approve, edit, or reject the next action.
  4. The action result is written back into task history.

CLI reference

The CLI gives you a faster way to attach projects, start task runs, inspect status, and export summaries.

terminal
otherone task create "Refactor authentication module"
otherone task status
otherone task approve next
otherone task export --format markdown

Configuration

Workspace behavior can be configured through a local otherone.config.json file.

otherone.config.json
{
  "workspace": {
    "name": "otherone-web",
    "memory": true
  },
  "tools": {
    "filesystem": "approval_required",
    "terminal": "approval_required",
    "browser": "read_only"
  },
  "review": {
    "requireApprovalForFileEdits": true,
    "requireApprovalForCommands": true
  }
}

API

Use the API to create tasks, inspect execution state, and integrate Otherone into internal workflows.

TypeScript
import { Otherone } from "@otherone/sdk";

const client = new Otherone({
  apiKey: process.env.OTHERONE_API_KEY
});

const task = await client.tasks.create({
  workspaceId: "ws_123",
  goal: "Review the auth module and propose safe patches.",
  reviewMode: "approval_required"
});

console.log(task.id);
API actions follow workspace permissions

Even when tasks are created through the API, tool permissions and review settings are still enforced by the workspace.
